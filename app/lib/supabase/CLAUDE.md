# app/lib/supabase/ — Supabase Client Factories

Three factories. Use the right one for the context or you'll either break SSR cookie handling or accidentally ship a service key to the browser.

## Which factory to use

| Context | Factory | File |
|---------|---------|------|
| Server components, route handlers, server actions | `createClient()` | `server.ts` |
| `"use client"` components (hooks, event handlers) | `createClient()` | `client.ts` |
| Admin operations that bypass RLS | `serviceRoleClient()` | `service-role.ts` |

Both `server.ts` and `client.ts` export a function named `createClient` — import from the right path. They're not interchangeable: `server.ts` reads Next.js cookies for SSR session hydration; `client.ts` uses a browser client.

`serviceRoleClient()` reads `SUPABASE_SERVICE_ROLE_KEY` which is server-only. Any `"use client"` file that imports `service-role.ts` is a bug — the key must never ship to the browser. Always call `requireAdmin()` from `app/lib/auth/require-admin.ts` before using the service role client.

## `types.ts` — regen procedure

`types.ts` is generated by `npm run gen:types` (run from `app/`). It contains hand-edited additions listed in a warning block near the top. After regenerating:

1. Read the warning block at the top of `types.ts`
2. Re-apply every listed hand-edit before committing
3. Commit type edits in their own PR if possible — avoids collision with the other dev

The hand-edit warning block lists columns that `gen:types` doesn't capture (e.g. view columns, columns added by migrations not yet reflected in the remote schema). If you add a new hand-edit, add it to the warning block.

## `cached.ts`

Wraps expensive read queries with Next.js `unstable_cache`. Only use this for data that doesn't change per-user (e.g. the film catalog). User-specific data should NOT be cached here — it bypasses RLS scoping.

## profiles column-level grants

`profiles` has column-level privileges layered on its RLS policies (mig 0203).
Never `.select("*")` on profiles with anon/authenticated clients: PostgREST
expands `*` to all columns and fails with `permission denied`. Use explicit
column lists from `app/lib/queries/profiles.ts`.

`unsubscribe_token` is service-role-only for reads. Client roles also cannot
update `must_change_password`, `role`, `is_starter`, `starter_order`, or
`email_added_at`.
